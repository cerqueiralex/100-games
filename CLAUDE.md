## Commands

Node is NOT on the default PATH on this machine (Flatpak VS Code sandbox). Prefix every npm/node command with:

```bash
export PATH="$HOME/.local/opt/node-v22.17.0-linux-x64/bin:$PATH"
```

- `npm run dev` — dev server on :5173, also exposed on the LAN (`--host`)
- `npm run build` — typecheck (`tsc --noEmit`) + production build to `dist/`
- `npm run check` — typecheck only
- `npm run validate` — integrity checks for game content: rebuilds every hand-authored crossword (intersections, accidental adjacent words), test-generates sudoku, Cryptogram, Logic Puzzles and Color Connect puzzles, checks the Word Wheel and Cryptogram content banks, and re-verifies the 50 newer games' generators and content banks — every guess-free deduction puzzle (nonogram line-solver, binary grid, skyscrapers, futoshiki, kakuro, mathdoku, killer sudoku, tents, aquarium, hashi/bridges, slitherlink, nurikabe, fleet finder) is re-proven to have a single deduction-only solution, the runtime generators (lights-out GF(2) par, sokoban, pipes, gridlock, tower-of-hanoi, untangle, tangram, jigsaw geometry, peg solitaire, klondike deals) and the word/number/logic content banks are all re-checked (seeded, deterministic, per-tier). **Run this after any edit to `src/games/crossword/logic/puzzles.ts`, `src/games/word-wheel/logic/levels.ts`, `src/games/cryptogram/logic/words.ts`, or any generator/puzzle-bank/content file the validate payloads cover.**

There is no test framework or linter; `npm run build` and `npm run validate` are the gates. The GitHub remote uses SSH (`git@github.com:cerqueiralex/100-games.git`) — the HTTPS URL has no credentials on this machine.

**Deployment**: every push to `main` triggers `.github/workflows/deploy.yml`, which validates, builds with `VITE_BASE=/100-games/`, and deploys `dist/` straight to Pages via `upload-pages-artifact` + `deploy-pages` → live at https://cerqueiralex.github.io/100-games/ (the user plays the installed PWA from there on iPhone). Because of subpath hosting, in-app URLs must use `import.meta.env.BASE_URL`, never absolute `/` paths. Two settings are load-bearing and were configured by hand (the workflow token lacks admin rights, so `actions/configure-pages` must NOT be added): Settings → Pages → Source = "GitHub Actions", and the `github-pages` environment's deployment-branch policy must allow `main` — if deploy runs fail in seconds with "Branch main is not allowed to deploy", that policy is the cause. The old gh-pages-branch publisher route is dead (its deploy step times out); don't revive it.

## What this is

An offline-first PWA of classic puzzle games (React 18 + Vite + TypeScript, no backend, no router — navigation is component state in `App.tsx`). Sixty-seven games so far: Sudoku, Crossword, Word Wheel, Memory Match, Simon, N-Back, Dual N-Back, Number Merge, Color Connect, Tic-Tac-Toe, Image Puzzle, Maze, Cryptogram, Minesweeper, Logic Puzzles, Battleship, House Puzzles, Nonograms, Killer Sudoku, Kakuro, Skyscrapers, Futoshiki, Binary Grid, Aquarium, Tents & Trees, Bridges, Slitherlink, Nurikabe, Lights Out, Fleet Finder, Code Breaker, 2048, Make 24, Target Number, Math Sprint, Magic Square, Sequence Cracker, MathDoku, Word Guess, Word Search, Word Ladder, Anagram Sprint, Missing Vowels, Hangman, Letter Hunt, Backwards Span, Number Trail, Pattern Recall, Moving Cups, Schulte Table, Stroop Match, Odd One Out, Count & Compare, Sokoban, Pipes, Gridlock, Tower of Hanoi, Tangram, Untangle, Laser Mirrors, Jigsaw, Peg Solitaire, Connect Four, Reversi, Checkers, Dots & Boxes, Klondike Solitaire. All persistence is localStorage under versioned keys (`100games.v1.*`) via `src/platform/storage.ts` — game-specific extras (e.g. Logic Puzzles preset progress) go through its `readGameData`/`writeGameData` helpers so `resetAll` (which sweeps the whole `100games.v1.` prefix) always finds them; game code never touches `localStorage` directly. It is used on desktop browsers and installed on iPhone via Safari's Add to Home Screen (PWA config in `vite.config.ts`).

Content notes: Word Wheel levels (`src/games/word-wheel/logic/levels.ts`) and the Cryptogram word bank + hidden answers (`src/games/cryptogram/logic/words.ts`; puzzles are assembled at runtime by `generateCryptoPuzzle`, letters disguised behind shuffled pictograms from `design/icons.tsx`) are hand-authored and covered by `npm run validate`, like crossword puzzles. Logic Puzzles (`src/games/logic-grid/`) has a propagation solver whose invariant everything relies on: a clue set is only accepted when sound deduction alone fully decides the grid — which proves both uniqueness and guess-free solvability. Its 25 presets are original content derived deterministically from baked seeds (`logic/presets.ts`) and re-verified by `npm run validate`; the theme bank (`logic/themes.ts`) is original too — never copy puzzles from other sites. House Puzzles (`src/games/house-puzzles/`) reuses the same guess-free invariant with a positional solver: houses in a row, ordering clues (at/adjacent/direct/somewhere/between), seeded generation re-verified by `npm run validate`; its category bank (`logic/themes.ts`) is original content. Color Connect and Maze generate levels at runtime with guaranteed solvability. Image Puzzle loads photos listed in `public/puzzles/manifest.json` — users add custom photos by dropping a file there and listing it in the manifest. Drag-driven games (Number Merge, Color Connect) use pointer events with `touch-action: none` and rect-math cell hit-testing, not `elementFromPoint`.

`backlog.txt` at the repo root is the user's personal scratch file — leave it untracked and don't delete it.

## Architecture: platform vs. games

The core invariant: **games live in isolated folders under `src/games/<id>/`; everything shared lives in `src/platform/`**. Games never import from each other (small helpers are duplicated rather than shared across game folders).

A game plugs in by exporting a `GameDefinition` (see `src/platform/types.ts`) from its `index.ts` and being added to the list in `src/platform/registry.ts`. That's the entire integration surface. `GameDefinition.category` is required and must come from the fixed vocabulary in `src/platform/categories.ts` (logic/words/memory/focus/numbers/spatial/strategy/reflex) — the home filter chips, card tags and profile category statistics all derive from it; if a future game truly fits none, add ONE entry there plus the `CategoryId` union, never a per-game string. `GameDefinition.tutorial` is required: every game ships an illustrated how-to-play as `tutorial.tsx` in its folder (3–6 `TutorialStep`s whose art composes the `.tut-*` CSS primitives — see the Tutorials section of DESIGN.md). The shell surfaces it on the setup screen and via the header help button. The platform's `GameShell` (`src/platform/components/GameShell.tsx`) then provides for free: difficulty selection (five tiers: easy/medium/hard/pro/extreme — every game must supply configs for all five; `Record<Difficulty, …>` maps make the compiler enforce it), assist toggles persisted per game, the timer (games with their own pre-game picker, like Logic Puzzles' mode menu, call `holdClock(true)` so the clock only starts at real play), pause (the board hides on real pause only — finished boards stay visible for review), restart/quit (recorded as `abandoned` unless the game finished or was saved), save & resume, the tutorial entry points, result recording, the closable completion modal with a "Show results" reopen pill, and the shareable win card.

Data flows one way: the shell passes `GameProps` down (`difficulty`, `assists`, `paused`, `elapsedSec`, `onToggleAssist`, plus `savedState`/`registerSnapshot` for save-resume); the game reports upward through `events.onStats(...)` (continuously, so abandons capture the latest state) and `events.onFinish(...)` (exactly once — games guard with a `done` ref). Games generate their board in a `useMemo`; the shell remounts them via a `key` to start a new game.

Save & resume is mandatory per game (see DESIGN.md "Save & resume"): register a JSON-serializable snapshot provider in a dependency-less effect, and hydrate every initializer from `savedState` when present. The shell stores one save per game (`100games.v1.saves`), shows the Continue card on setup, restores elapsed time, and deletes the save when its session finishes.

### Assist/help tracking (user-mandated, preserve it)

Every game result stores both `assistsEnabled` and `assistsUsed`; `cleanWin` = won with zero hints and zero assists used. Convention: **passive** assists (highlights, auto-check, slow pace) are added to `assistsUsed` whenever enabled — including when toggled on mid-game; **active** assists (hint/peek/reveal/replay buttons) are added on use and usually also increment `hintsUsed`. History and statistics distinguish clean wins from assisted wins everywhere — new features must not blur this.

## Design system (mandatory)

`src/platform/design/DESIGN.md` is the rulebook for ALL UI work; the user has explicitly required it be followed. The load-bearing rules:

- Colors only via tokens in `design/tokens.css`. Surface themes (`data-theme`: black/dim/light) and accent themes (`data-accent`: orange default, blue/green/red/purple/white) are attributes on `<html>` set by `AppState`. The accent recolors every tool in every game; semantic colors (`--good`, `--bad`) never follow the accent.
- Surfaces are **flat with NO gradients**, defined once in `design/effects.css` and bound both to `.fx-card` and to every shared card-component class listed there — component CSS never declares its own card background/border (this is what keeps Settings etc. in sync with redesigns). The depth standard is the **extruded "candy/pushable" style**: flat fills with a darker inset bottom edge (`box-shadow: inset 0 -3/4px 0 var(--edge)`, token per theme; solid colored fills use `rgba(0,0,0,~0.25)` instead) on cards, buttons, and gap-separated game tiles. Buttons press DOWN: `:active` = `translateY(2px)` + edge shrinks to 1px — but never add a press transform to elements positioned via inline transforms (e.g. Word Wheel letters). NO drop shadows on resting surfaces (floating overlays like modals/pills keep `var(--shadow)` for elevation), no specular edges, no 3D tilt. Deliberate flat opt-outs: home search bar, in-game info strip, continuous boards (sudoku/crossword/logic-grid cells).
- Charts (profile page) are dependency-free SVG components in `platform/components/charts.tsx`, colored per game from the `--play-*` content palette (via the chart-safe slot list in `charts.tsx` — never `--play-9`, which is white).
- Tile boards must keep stable geometry (DESIGN.md "Tile grids"): pin both grid axes (`grid-auto-rows: 1fr` or `aspect-ratio` on the tile), and tile state/press styles may change paint only (colors, transform) — never border-width, padding, or font-size.
- Icons only from `design/icons.tsx` — monochrome SVG, `currentColor`. **No emojis in UI controls** (emojis are fine as game content, e.g. memory-card faces, and celebratory content). Game identity icons are the one exception: colorful sticker-style SVGs in `design/gameIcons.tsx` (keyed by game id, fixed own palette, documented token opt-out) shown on the home cards — a new game adds its sticker there.
- In-game toolbars are rows of `PadTool` buttons (`platform/components/ui`) in equal-width grids (`grid-auto-flow: column; grid-auto-columns: 1fr`). `PadTool` is the standard for every game tool: accent-tinted with hover/press effects and a built-in click sound (`silent` prop when the handler plays its own sfx, `active` for toggles) — never hand-write `<button className="pad-tool">`.
- **Bottom menu is ONE card**: every game puts ALL its bottom controls — tool rows AND any input surface (keyboard/d-pad/digit pad) — inside a single `.game-tools fx-card`, never sibling panels. It docks to the viewport bottom (`position: sticky`) so controls stay reachable while a tall board scrolls behind it; as a floating overlay it carries `var(--shadow)` and sits at `z-index: 30` — the pause overlay (z 40) and all modals/pills layer above it. A game with no tools (or whose tools are all assist-gated off) renders no card at all. On phone widths (`max-width: 700px`) the card must stay around a third of the viewport or less so the board keeps the screen: oversized content compacts (Word Wheel's wheel) or scrolls internally (Logic Puzzles' clue list) — see DESIGN.md.
- All CSS lives in `src/styles/global.css` (game sections are marked); tokens/themes live in `design/tokens.css`, imported at its top.

## Game content notes

- Sudoku puzzles are generated at runtime with a uniqueness-guaranteeing solver (`src/games/sudoku/logic/generator.ts`).
- Crossword puzzles are hand-authored coordinate/word lists; the engine derives and validates the grid. `validatePuzzle` rejects mismatched intersections and any letter run without a matching entry — this is what `npm run validate` runs.
- Timed/sequence games (Simon, N-Back, Dual N-Back) drive trials with timeout chains inside effects keyed on `paused` — pausing clears timers and resuming replays the current sequence/trial rather than resuming mid-flight. Battleship's enemy turn uses the same pattern, and its AI is deliberately stateless (each shot is derived from the visible board in `logic/ai.ts`), which is what makes save/resume free.
- The new deduction puzzles (Nonograms, Binary Grid, Skyscrapers, Futoshiki, Kakuro, MathDoku, Killer Sudoku, Tents & Trees, Aquarium, Bridges/hashi, Slitherlink, Nurikabe, Fleet Finder) all reuse Logic/House Puzzles' guess-free invariant: each ships a propagation solver (Nonograms is a line-solver; Fleet Finder's is `logic/solver.ts`'s `propagateFix`) and its seeded generator only accepts a board that sound deduction alone fully decides — which proves both uniqueness and guess-free solvability. `npm run validate` re-proves every tier; never author or ship a board the solver can't decide.
- Lights Out ships an **exact** GF(2) minimal-press par: Gauss–Jordan over the toggle matrix (minimum-weight null-space search where the matrix is singular — 4×4 nullity 4, 5×5 nullity 2). It is computed live per state (sub-ms), drives the scoring par and the par-meter/hint assists, and must stay exact — no approximate press counts.
- Gridlock (Rush Hour) ships a **pre-baked, BFS-verified puzzle bank** (`src/games/gridlock/logic/puzzles.ts`, banded by tier) picked deterministically by seed; live full-component BFS is tens of seconds on hard tiers and would freeze the `useMemo`, so the bank MUST stay baked (regenerate offline via the bake harness, never hand-edit). Sokoban instead guarantees solvability by **reverse-pull construction** (pull crates from the solved state; the flipped move list is a legal forward solution) and sets par with a push-optimal A*.
- Tower of Hanoi computes par with **Frame–Stewart** DP on the 4-peg extreme tier (the `2^n−1` closed form only holds for 3 pegs); its hint is a BFS over the (tiny) state graph so it returns the true best move from any position, not a from-start replay.
- Strategy games (Connect Four, Reversi, Checkers, Dots & Boxes) are **minimax + alpha-beta** vs-AI on fast numeric board cores, and reuse Tic-Tac-Toe's vs-Robot / vs-Friend (pass-and-play) shape: best-of-N rounds, sudden death, `holdClock(true)` on the pre-game menu, and `hideStats`/headline/subline result presentation for local-multiplayer wins.
- Klondike and Peg Solitaire ship **baked winnable/solvable start banks** because their DFS solvers are too slow to run at mount: Klondike draws easy/medium from a proven-winnable seed bank (harder tiers use random deals + unlimited undo), Peg Solitaire's pro/extreme starts come from solver-verified lists. Jigsaw reuses the Image Puzzle photo manifest (`public/puzzles/manifest.json`) via `import.meta.env.BASE_URL` through its own loader (no cross-game import), with a `--play-*` gradient fallback when a photo 404s.

## Verification workflow used here

UI changes are verified by running the dev server and driving the app with `puppeteer-core` (already a devDependency; connect with `browserURL`, no browser download) against the host machine's headless Chrome (`host-spawn google-chrome --headless --remote-debugging-port=9222 --user-data-dir=/tmp/... about:blank`). The sandbox's `/tmp` is not shared with the host — write screenshots to a path under `$HOME`. Games pick puzzles with `Math.random`; stub it via `page.evaluateOnNewDocument` when a deterministic board is needed (e.g. auto-solving a crossword to reach the win screen).
